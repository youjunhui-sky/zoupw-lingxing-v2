# 领星 ERP 数据接入与 AI 智能看板项目方案

> 文档版本:v1.0
> 编写日期:2026-05-26
> 编写人:技术服务团队

---

## 一、项目背景

客户为跨境电商公司,日常运营数据沉淀在「领星 ERP」中,涉及销售、订单、库存、广告、财务等多个模块。当前痛点:

1. **数据散落在领星 ERP 界面,老板/运营需要登录系统逐个模块查看**,缺乏统一视图。
2. **核心指标无主动推送机制**,异常情况(库存断货、广告 ACOS 飙升等)发现滞后。
3. **历史数据沉淀利用率低**,未形成数据资产,难以做趋势分析与决策支持。
4. **多店铺/多站点场景下汇总效率低**,依赖人工导 Excel。

客户希望我们提供「数据接入 → 指标沉淀 → 看板呈现 → 主动推送」的全流程 AI 服务。

---

## 二、项目目标

| 目标 | 衡量标准 |
|---|---|
| **数据自动接入** | 领星核心模块数据 T+0 / T+1 自动同步入库,0 人工干预 |
| **统一看板呈现** | 一个入口看完所有核心指标,支持多维度筛选 |
| **核心指标主动推送** | 日报/周报/异常告警通过飞书自动推送,触达率 100% |
| **数据资产沉淀** | 历史数据可追溯、可二次分析,为后续 AI 应用(智能补货、广告优化)打基础 |

## 三、整体架构

```
┌──────────────────────────────────────────────────────────────┐
│                      领星 OpenAPI                            │
│         (订单/库存/广告/财务/Listing/Review 等)              │
└─────────────────────────────┬────────────────────────────────┘
                              │ OAuth 授权 + 增量拉取
                              ▼
┌──────────────────────────────────────────────────────────────┐
│   数据采集层 (systemd timer + APScheduler in-process)        │
│   - 全量初始化任务   - 增量同步任务   - 失败重试 + 告警      │
└─────────────────────────────┬────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────┐
│         数据存储层  (PostgreSQL 16 + Redis 缓存)             │
│   ODS 原始层 → DWD 明细层 → DWS 汇总层 → ADS 指标层         │
└─────────────────────────────┬────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────┬─────────────────┐
│         看板呈现层 (Metabase / 飞书多维)   │  推送层 (飞书)  │
│   - 销售看板  - 库存看板  - 广告看板       │  - 日报机器人   │
│   - 财务看板  - 自定义看板                 │  - 周报机器人   │
│                                            │  - 异常告警     │
└────────────────────────────────────────────┴─────────────────┘
```

### 技术选型说明

| 层 | 选型 | 理由 |
|---|---|---|
| 采集 | Python 3.11 + APScheduler + httpx | 领星 SDK 成熟、易维护;调度灵活;并发性能足够 |
| 存储 | PostgreSQL 16 | 开源、JSON 字段支持好、聚合性能优于 MySQL;后期可平滑升级至 TimescaleDB |
| 缓存 | Redis 7 | 缓存高频查询 + API 限流计数器 |
| 看板 | Metabase(开源)+ 飞书多维表格 | Metabase 拖拽式 BI 零开发;飞书多维表格便于客户运营自助使用 |
| 推送 | 飞书自定义机器人 + 卡片消息 | 客户已在飞书生态、富文本卡片表现力强、零成本 |
| 部署 | systemd unit + systemd timer(小规模)/ K8s(大规模)| 零额外依赖、运维交接最简;规模化时再上 K8s |

## 四、数据接入范围

### 部署层补充说明

本项目采用 `systemd` 部署,具体分工:

- **`lingxing-erp.service`** - 常驻进程,同时承载 `uvicorn`(FastAPI 出看板 API)与 `APScheduler` 全部定时任务(订单/库存/广告/日报)
  - `main.py` 的 `lifespan` 中构造 `SyncContext`(共享 engine / redis / client)并 `await scheduler.start()` -- **uvicorn 与 APScheduler 真同进程**
  - 关闭时 `await scheduler.stop()` → `client.close()` → `redis.aclose()` → `engine.dispose()` 顺序释放
- **`lingxing-erp.timer`** - systemd timer 兜底触发器,APScheduler 内部失败时由 timer 强制重启 service
- **`lingxing-erp.target`** - 聚合 unit,管理启停顺序

**为什么不上 Docker Compose / K8s**:

1. 本项目无 Celery / RabbitMQ 等多 worker 进程,单进程足以支撑(店铺数 ≤ 几十、日订单 < 10 万的跨境电商客户)
2. 客户现场"网络/服务器环境差"是已识别风险,Docker 引擎是额外故障点
3. 节省运维交接成本 -- 客户运维只需懂 `systemctl`,无需懂容器编排

**部署最小资源**:2C4G 云主机 + Ubuntu 22.04/24.04 即可，后续规模化可平滑迁 K8s(业务代码无需改动)。

### 3.1 Secret 加密存储

为避免 `.env` 明文泄露（git 误推、服务器被脱裤），项目对**8 个敏感字段**强制 Fernet 对称加密:

- `LX_APP_SECRET` / `LX_DOC_SECRET`（领星开放平台）
- `POSTGRES_PASSWORD`（数据库密码）
- `FEISHU_DAILY_WEBHOOK` / `FEISHU_ALERT_WEBHOOK` / `FEISHU_WEEKLY_WEBHOOK`（飞书机器人）
- `FEISHU_APP_SECRET` / `FEISHU_BITABLE_APP_TOKEN`（飞书自建应用）

**加密格式**:`enc:<fernet_token>`(前缀显式区分密文/明文)

**Key 管理**:
1. 生成:`python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`
2. 注入:通过 `LINGXING_ENC_KEY` 环境变量(systemd `EnvironmentFile=` / K8s Secret)
3. **不进仓库、不可泄漏**

**运维流程**:
```bash
export LINGXING_ENC_KEY=<key>
python scripts/encrypt_env.py        # 自动加密默认 8 个字段, .env → .env.bak
```

**业务透明**:`Settings` 加载时通过 `field_validator` 自动解密,业务代码继续用 `settings.lx_app_secret` 等标准访问,零侵入。

**失败行为**:
- 缺 key + 明文 → WARNING 后原样返回(开发态友好)
- 缺 key + 密文 → **RuntimeError**(防静默拿到无意义 `enc:...`)
- key 不匹配 → **RuntimeError**(防篡改/防错配)

详见 `src/lingxing/config/secrets.py` 与 `scripts/encrypt_env.py`。

### 4.1 一期接入模块(MVP,2-3 周完成)

| 模块 | 关键数据 | 同步频率 |
|---|---|---|
| **店铺基础** | 店铺、市场、品牌、SKU 主数据 | 每日 1 次 |
| **订单** | 订单头、订单明细、订单状态流转 | 每 1 小时增量 |
| **销售汇总** | 日销售额、订单量、退款 | 每 1 小时 |
| **库存** | FBA 库存、本地仓库存、在途库存 | 每 2 小时 |
| **广告** | Campaign / Ad Group 花费、曝光、点击、ACOS | 每日 1 次(T+1) |

### 4.2 二期扩展模块(按需)

- Listing 健康度(Buy Box、评分、Review)
- 财务(FBA 费用、回款、利润)
- 客服(邮件、Case、差评预警)
- 竞品监控、关键词排名

## 五、核心指标体系

### 5.1 销售指标
- GMV(日/周/月/同环比)
- 订单数、客单价、退款率、净销售额
- 各店铺/市场销售排行 Top N
- SKU 销售排行 Top 20

### 5.2 流量指标
- Session、PageView、转化率
- Buy Box 占有率
- 流量来源构成

### 5.3 广告指标
- ACOS、TACoS、ROAS
- CPC、CTR、CVR
- 广告占比销售额、广告利润贡献

### 5.4 库存指标
- 可售天数(DOI)
- 缺货 SKU 数 / 滞销 SKU 数
- 补货建议清单(基于销售速率 + 头程时长)

### 5.5 财务指标
- 毛利、毛利率
- FBA 仓储费、操作费占比
- 回款周期、资金占用

## 六、飞书推送方案

### 6.1 推送通道
- **群机器人**:每个部门/团队一个群,机器人定时投放
- **卡片消息**:富文本 + 表格 + 跳转按钮 + @相关人

### 6.2 推送内容设计

| 推送类型 | 频率 | 内容 | 接收人 |
|---|---|---|---|
| **每日早报** | 工作日 9:00 | 昨日 GMV、订单、广告 ACOS、库存预警 Top 5 | 老板群、运营群 |
| **每周复盘** | 周一 10:00 | 上周整体表现、环比变化、Top SKU、问题清单 | 管理层群 |
| **库存告警** | 实时(每 2 小时检测) | 可售天数 < 15 天的 SKU @采购负责人 | 采购群 |
| **广告异常告警** | 实时(每小时检测) | ACOS 飙升 50%+ 的广告活动 @广告投手 | 广告群 |
| **大额退款告警** | 实时 | 单笔退款 > 阈值 | 客服群 |

### 6.3 看板嵌入飞书
- Metabase 看板生成嵌入链接 → 嵌入飞书文档/工作台
- 运营点击飞书消息卡片可直接跳转到对应看板
- 飞书多维表格作为「轻看板」补充,支持运营自助筛选

## 七、项目分期与交付

### 7.1 MVP 阶段(第 1-3 周)

| 周次 | 交付物 |
|---|---|
| 第 1 周 | 领星 OAuth 接入、数据库设计、订单/销售模块同步 |
| 第 2 周 | 库存/广告模块同步、Metabase 部署、3 个核心看板 |
| 第 3 周 | 飞书日报机器人、库存告警机器人、客户验收 |

**MVP 验收标准**:客户能在飞书每日收到核心指标、能在 Metabase 看 3 个看板(销售/库存/广告)。

### 7.2 完整版(第 4-8 周)

- 全量模块接入(财务、Listing、Review)
- 完整指标体系 + 自定义看板
- 飞书多维表格交付(运营自助查询)
- 异常告警规则完善
- 文档 + 运维交接

### 7.3 后续可扩展(AI 增值服务)

- 智能补货建议(基于历史销售 + 季节性 + 头程预测)
- 广告优化建议(基于 ACOS 趋势 + 关键词表现)
- 评论情感分析与差评预警
- 业绩异动归因分析(销量下滑原因 AI 解读)

## 九、风险与应对

| 风险 | 应对 |
|---|---|
| 领星 API 限频、字段变更 | 增量拉取 + 退避重试 + 字段变更告警;落库后看板不依赖实时 API |
| 客户多店铺授权复杂 | 一店一授权，授权 Token 加密存储（Fernet 对称加密，详见 §部署层补充说明），过期自动刷新 |
| 指标口径与客户理解不一致 | MVP 前必须签字确认指标定义文档 |
| 客户网络/服务器环境差 | 提供 systemd 部署脚本 + 离线 RPM/deb 包两种方案 |
| 数据量增长导致查询慢 | 预留 TimescaleDB / ClickHouse 升级路径 |
